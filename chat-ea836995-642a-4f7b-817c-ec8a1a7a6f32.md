# ALEXA Bank Account Opening — Low-Level Design & API Strategy

This is a well-scoped assignment. Let me give you a thorough design breakdown.

---

## 1. Core Domain Model

Think of the account opening process as a **state machine** for a single `Registration` entity:

```
INITIATED → KYC_COMPLETE → SUBMITTED → (CONFIRMED)
     ↑            |
     +-- PAUSED --+
```

| State | Meaning |
|---|---|
| `INITIATED` | Customer started; Name, Address, DOB saved (minimum for pause) |
| `PAUSED` | Customer explicitly paused after providing minimum KYC |
| `KYC_COMPLETE` | All mandatory KYC fields provided and valid |
| `SUBMITTED` | All data (KYC + optional) submitted; confirmed with a request ID |

---

## 2. API Design (Resource-Oriented)

I'd recommend **four endpoints** on a single resource `registrations`:

### A. Start Registration (with minimum KYC)
```
POST /api/v1/registrations
```
- **Purpose:** Create a new registration with at least Name, Address, Date of Birth.
- **Body:** `{ name, address, dateOfBirth }`
- **Validations:** NL postcode format (`1234 AB`), DOB not in future, name non-blank.
- **Response:** `201 Created` — returns `registrationId` + status `INITIATED`.
- **Why separate?** The assignment says the customer must provide name, address, DOB *first*, and then can pause. This naturally maps to creation.

### B. Field-Level Validation (instant feedback)
```
POST /api/v1/registrations/validate
```
- **Purpose:** Validate a *single field* or a *subset of fields* on-the-fly, before any save.
- **Body:** `{ fieldName: "postCode", value: "1234 AB" }` (or a map of fields)
- **Response:** `200 OK` with `{ valid: true/false, errors: [...] }`
- **Why?** The requirement says *"front end must be able to validate each field as soon as possible."* This gives the UI real-time per-keystroke validation without persisting anything.

### C. Update Registration (add remaining details)
```
PATCH /api/v1/registrations/{registrationId}
```
- **Purpose:** Incrementally add KYC fields (ID document, account type) and optional fields (salary, email, etc.).
- **Body:** Partial JSON — only the fields being added/updated.
- **Validations:** Field-level validation runs on every patch; state transitions accordingly.
- **Response:** `200 OK` — returns updated registration + current status.
- **Why PATCH not PUT?** The process is incremental; the customer fills in fields over multiple sessions.

### D. Submit Registration (final confirmation)
```
POST /api/v1/registrations/{registrationId}/submit
```
- **Purpose:** Final submission. All mandatory KYC fields must be present and valid.
- **Validations:** Checks completeness — if anything is missing, returns `400` with details.
- **Response:** `200 OK` — returns `requestId` (confirmation) + status `SUBMITTED`.
- **Why separate from PATCH?** Explicit intent. The "submit" is a business action that triggers the *"once accepted, it must be honoured"* guarantee.

### Pause & Resume

Pause is **implicit** — once the `POST` (creation) is done, the registration simply sits in `INITIATED` state. The customer can come back and `PATCH` more data whenever they want. If you want an explicit pause:

```
POST /api/v1/registrations/{registrationId}/pause
```

Resume is just the next `PATCH` call. The `GET` below lets the UI reload the form state:

```
GET /api/v1/registrations/{registrationId}
```

---

## 3. Data Model (Entity Design)

A single `Registration` table/entity:

| Column | Type | Notes |
|---|---|---|
| `id` (PK) | UUID | Generated on creation |
| `request_id` | String | Generated on successful submit |
| `status` | Enum | `INITIATED, PAUSED, KYC_COMPLETE, SUBMITTED` |
| `name` | String | Mandatory KYC |
| `street_name` | String | Mandatory |
| `house_number` | String | Mandatory |
| `post_code` | String | NL format: `\d{4}\s[A-Z]{2}` |
| `city` | String | Mandatory |
| `date_of_birth` | LocalDate | Mandatory, must be ≥ 18 |
| `id_document` | String | Mandatory KYC (could be document number) |
| `account_type` | Enum | `SAVINGS, CURRENT, INVESTMENT` |
| `starting_balance` | BigDecimal | Optional |
| `monthly_salary` | BigDecimal | Optional |
| `interested_in_other_products` | Boolean | Optional |
| `email` | String | Optional, validate format |
| `created_at` | Timestamp | Audit |
| `updated_at` | Timestamp | Audit |

Use an **embedded `Address` value object** to keep things clean.

---

## 4. Validation Strategy (Layered)

This is key to the assignment. Use **two layers**:

### Layer 1: Field-level (via `/validate` and on every request)
- Use **Bean Validation annotations** (`@Pattern`, `@NotBlank`, `@Email`, custom `@NLPostCode`).
- Write a **custom validator** for NL postcode: regex `^\d{4}\s[A-Z]{2}$`.
- Age validation: custom `@MinAge(18)` on `dateOfBirth`.

### Layer 2: Completeness check (on `/submit`)
- A **service-level check** that all mandatory KYC fields are non-null.
- This is *not* annotation-based — it's business logic in the service, because during `PATCH` you deliberately allow partial data.

**Key insight:** Use *different DTOs* (or validation groups) for creation vs. update vs. submit:
- `CreateRegistrationRequest` — validates Name, Address, DOB (all required).
- `UpdateRegistrationRequest` — all fields optional, but if present, validate format.
- On submit — service checks completeness programmatically.

## 6. Error Handling

Use a global `@RestControllerAdvice` that returns a consistent error body:

```json
{
  "timestamp": "2026-02-12T10:00:00Z",
  "status": 400,
  "errors": [
    { "field": "postCode", "message": "Must be 4 digits, a space, and 2 uppercase letters" },
    { "field": "dateOfBirth", "message": "Customer must be at least 18 years old" }
  ]
}
```

Map `MethodArgumentNotValidException` → structured field errors.
Map `RegistrationNotFoundException` → `404`.
Map illegal state transitions (e.g., submit when KYC incomplete) → `409 Conflict`.

Customer                    API                         DB
   |── POST /registrations ──▶ validate name/addr/dob ──▶ save (INITIATED)

   |◀── 200 ─────────────────────────┘
   |

   |◀── 200 {requestId: "REQ-20260212-XXXX"} ─────┘
```

## 10. Key Design Decisions to Call Out in Your Docs

1. **Why a multi-step API instead of a single POST?** — Enables pause/resume and real-time validation per the requirements.
2. **Why a separate `/validate` endpoint?** — Decouples UI feedback from persistence; the frontend can validate without creating a record.
3. **Why PATCH and not PUT?** — Partial updates are the natural model for incremental form filling.
4. **Why a separate `/submit`?** — Makes the "once accepted, must be honoured" guarantee explicit and intentional.
5. **Idempotency** — Consider making submit idempotent (re-submitting returns the same `requestId`).

---

│ -- KYC: Account ───────────────────────────────  │

```

### Why this works
- **No joins** — everything the service needs is in one row.
- **Nullable columns** naturally model the incremental fill-in (pause/resume).
- **`request_id`** stays `null` until the customer submits — then it gets generated (e.g., `REQ-20260212-A3F8`).
- **`version`** column enables optimistic locking — prevents conflicts if the same registration is updated from two sessions.

---

## Column Design Decisions

| Column | Rationale |
|---|---|
| `id` (UUID) | Don't expose auto-increment IDs — UUIDs are safer in URLs and not guessable |
| `request_id` | This is the **customer-facing** confirmation ID, generated only on submit |
| `status` | Store as `VARCHAR`, map to Java enum — avoids DB-enum migration headaches |
| `post_code` | `VARCHAR(7)` — exactly fits `"1234 AB"` (4 digits + space + 2 letters) |
| `starting_balance` / `monthly_salary` | `DECIMAL(15,2)` — **never use float/double for money** |
| `created_at` / `updated_at` | Use JPA's `@CreationTimestamp` and `@UpdateTimestamp` |

# Separate DTOs — Absolutely

Use **separate DTOs** for each endpoint. Here's why and how:

---

## The Problem with One DTO

If you use a single DTO for both `POST` and `PATCH`:

- **`POST` requires** name, address, DOB → you'd mark them `@NotBlank`
- **`PATCH` allows** any subset of fields → everything must be nullable

These are **contradictory validation rules** on the same class. You'd end up hacking around it with validation groups or `if` checks in the service — messy.

---

## Recommended DTO Structure

```
dto/
├── request/
│   ├── CreateRegistrationRequest    ← POST /registrations
│   ├── UpdateRegistrationRequest    ← PATCH /registrations/{id}
│   ├── FieldValidationRequest       ← POST /registrations/validate
│   └── AddressRequest               ← shared, embedded in Create/Update
│
└── response/
    ├── RegistrationResponse         ← returned by all endpoints
    ├── ValidationResponse           ← returned by /validate
    └── ErrorResponse                ← returned by exception handler
```

---

## What Each DTO Carries

### `CreateRegistrationRequest`
Only the **minimum fields** needed to start (and to allow pause):

| Field | Constraint |
|---|---|
| `name` | `@NotBlank` |
| `address` | `@NotNull`, `@Valid` (nested) |
| `dateOfBirth` | `@NotNull`, `@Past`, custom `@MinAge(18)` |

`AddressRequest` (embedded):

| Field | Constraint |
|---|---|
| `streetName` | `@NotBlank` |
| `houseNumber` | `@NotBlank` |
| `postCode` | `@NotBlank`, `@Pattern(^\d{4}\s[A-Z]{2}$)` |
| `city` | `@NotBlank` |

### `UpdateRegistrationRequest`
**Everything is optional**. Only validate format *if present*:

| Field | Constraint |
|---|---|
| `name` | `@Size(min=1)` if provided |
| `address` | `@Valid` if provided |
| `dateOfBirth` | `@Past` if provided |
| `idDocument` | `@Size(min=1)` if provided |
| `accountType` | valid enum if provided |
| `startingBalance` | `@PositiveOrZero` if provided |
| `monthlySalary` | `@PositiveOrZero` if provided |
| `interestedInOtherProducts` | boolean, no validation needed |
| `email` | `@Email` if provided |

> **Key:** All fields are nullable. No `@NotNull` / `@NotBlank` anywhere. The completeness check only happens at `/submit` time in the service layer.

### `FieldValidationRequest`
For the real-time per-field validation endpoint:

| Field | Constraint |
|---|---|
| `fieldName` | `@NotBlank` — e.g., `"postCode"`, `"email"` |
| `value` | `@NotNull` — the raw value to validate |

### `RegistrationResponse`
**Single response DTO** for all endpoints (this one *can* be shared):

| Field |
|---|
| `id` |
| `requestId` (null until submitted) |
| `status` |
| All KYC + optional fields |
| `createdAt`, `updatedAt` |

---

## Why This Works Perfectly

| Concern | How it's solved |
|---|---|
| `POST` has strict validations | `CreateRegistrationRequest` has `@NotNull` / `@NotBlank` |
| `PATCH` allows partial data | `UpdateRegistrationRequest` has everything nullable |
| Submit needs completeness check | Done in **service layer**, not DTO annotations |
| Real-time field validation | Dedicated `FieldValidationRequest` DTO |
| Response consistency | One `RegistrationResponse` everywhere |

---

## Quick Rule of Thumb

> **Request DTOs → one per action** (they have different validation rules)
> **Response DTOs → share when the shape is the same**

Three request DTOs, one response DTO. Clean, testable, and each DTO does exactly one job.

# PUT is Totally Fine — Here's How to Adapt

You can absolutely use `PUT`. You just need to shift your mental model from *"send only what changed"* to *"send the full current state every time"*.

---

## How the Flow Changes

### With PATCH (partial)
```
POST   /registrations           → { name, address, dob }
PATCH  /registrations/{id}      → { idDocument }              ← only new fields
PATCH  /registrations/{id}      → { accountType, email }      ← only new fields
POST   /registrations/{id}/submit
```

### With PUT (full replacement)
```
POST   /registrations           → { name, address, dob }
PUT    /registrations/{id}      → { name, address, dob, idDocument }              ← full state
PUT    /registrations/{id}      → { name, address, dob, idDocument, accountType, email }  ← full state
POST   /registrations/{id}/submit
```

The frontend always sends the **complete registration** on every `PUT`. It first does a `GET` to load current state, merges in the new fields, and sends it all back.

---

## Impact on Your DTOs

This is where it actually gets **simpler** — you need fewer DTOs:

```
dto/
├── request/
│   ├── CreateRegistrationRequest    ← POST  (name, address, dob only)
│   ├── UpdateRegistrationRequest    ← PUT   (all fields, KYC mandatory)
│   ├── FieldValidationRequest       ← POST  /validate
│   └── AddressRequest               ← shared, embedded
│
└── response/
    ├── RegistrationResponse
    ├── ValidationResponse
    └── ErrorResponse
```

### `CreateRegistrationRequest` — stays the same
Only minimum fields to start:

| Field | Constraint |
|---|---|
| `name` | `@NotBlank` |
| `address` | `@NotNull`, `@Valid` |
| `dateOfBirth` | `@NotNull`, `@Past`, `@MinAge(18)` |

### `UpdateRegistrationRequest` — this is what changes

Since `PUT` means full replacement, you have **two zones**:

| Field | Constraint | Why |
|---|---|---|
| `name` | `@NotBlank` | Already provided during POST, must stay |
| `address` | `@NotNull`, `@Valid` | Same |
| `dateOfBirth` | `@NotNull`, `@Past` | Same |
| `idDocument` | **nullable** | Customer may not have filled this yet |
| `accountType` | **nullable** | Same |
| `startingBalance` | **nullable** | Optional field |
| `monthlySalary` | **nullable** | Optional field |
| `interestedInOtherProducts` | **nullable** | Optional field |
| `email` | `@Email` if provided | Optional field |

> **Key rule:** Fields that were already mandatory at creation time stay mandatory in `PUT`. Fields the customer hasn't reached yet remain nullable.

---

## PUT vs PATCH — Trade-offs

| Aspect | PUT | PATCH |
|---|---|---|
| **Simplicity of contract** | ✅ Easier to reason about — always full state | Partial payloads can be ambiguous |
| **Frontend burden** | ❌ Must always send everything | Only sends what changed |
| **Null ambiguity** | ❌ Is `null` "clear this field" or "not sent"? | No ambiguity — absent = don't touch |
| **Validation clarity** | ✅ You can enforce mandatory fields on every call | Harder — everything must be optional |
| **Bandwidth** | Slightly more | Slightly less |

---

## Handling the Null Ambiguity Problem

This is the **one gotcha** with `PUT`. If the customer sends:

```json
{ "name": "Jan", "address": {...}, "dateOfBirth": "1990-01-01", "email": null }
```

Does `email: null` mean *"I want to clear my email"* or *"I haven't filled it in yet"*?

**For this assignment, it doesn't matter** — both interpretations result in the same behavior (store null). But in production systems this is exactly why people prefer `PATCH`.

**Simple rule for your service:** On `PUT`, just overwrite every field with whatever was sent. Null means empty.

---

## My Recommendation

> **Stick with `PUT` if you prefer it.** For this assignment scope, the difference is minimal. The evaluator cares more about **consistency** — pick one approach, apply it correctly, and document *why* in your README.

Just make sure your `GET /registrations/{id}` returns the full current state so the frontend can load → merge → `PUT` back cleanly.
